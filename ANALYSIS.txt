Cube Project Analysis
=====================

Scope
- Standalone cube app only.
- Entry point: cube/index.php
- Runtime script: cube/public/js/dd.js

Current state
- The project structure is clear for a small graphics demo.
- Scene code, matrix helpers, canvas drawing, and object models are separated.
- The recent rotation change already improved the hottest animation path.

Main findings
1. Resize handling is broken.
   - Only canvasTop is resized.
   - canvasGrid and canvasAir keep their old dimensions.
   - Canvas.resizeCanvas updates object fields, but not the actual <canvas> element width and height.

2. Number-key object selection is broken.
   - dd.js uses String.fromCharCode(ev.key), but ev.key is already a string.
   - That makes parseInt unreliable for selecting the active object.

3. The standalone cube app still uses root-level image URLs.
   - dd.js loads /public/images/air.webp and /public/images/ground.png.
   - For a standalone /cube/ app, these should be relative to cube.

4. Matrix angle normalization is brittle.
   - Matrix.m_make_rx / ry / rz only subtract 360 once.
   - Negative angles or large angles can index outside the precomputed sin/cos arrays.

Fix order
1. Repair canvas resize behavior in Canvas.js and dd.js. Done.
2. Repair object selection from number keys in dd.js. Done.
3. Make image paths standalone-safe in dd.js. Done.
4. Harden angle normalization in Matrix.js. Done.

Notes
- Keep changes minimal.
- Preserve current visual behavior.
- Validate after each edit slice.

Applied changes
- Canvas.js now keeps a reference to the real canvas element and updates its width and height during resize.
- dd.js now resizes canvasTop, canvasGrid, and canvasAir together.
- dd.js now parses number keys directly from ev.key.
- dd.js now uses relative image paths for the standalone cube app.
- Matrix.js now normalizes any degree input safely into the 0..359 range.

Remaining recommendation
- Run a manual browser check for resize behavior and layer alignment, since this review used static validation only.

Second pass findings
1. Projection math does not use aspect ratio.
   - dd.js computes an aspect value, but Matrix.m_projection is called without it.
   - That means wide and tall windows can stretch the scene.

2. Resize does not rebuild projection data.
   - The canvas sizes now update correctly.
   - But dd.js does not recompute docWidth, docHeight, aspect ratio, or projectionMatrix on resize.
   - So a resized window can still render with stale camera math.

3. Canvas.drawRectangle has surprising behavior.
   - After drawing a rectangle, it also draws and fills a large circle at the origin.
   - That is probably leftover debug code and makes the method unsafe as a general helper.

4. Rotation helper order is inconsistent in utils.js.
   - RotateX multiplies M * R.
   - RotateY and RotateZ multiply R * M.
   - This is a latent math bug if those helpers are expected to behave consistently.

Second pass recommendation order
1. Fix projection/aspect handling in Matrix.js and dd.js. Done.
2. Recompute projection data inside resizeCanvas(). Done.
3. Remove the extra circle draw from Canvas.drawRectangle().
4. Normalize multiplication order in utils.js helper rotations.

Second pass applied
- Matrix.m_projection now accepts an aspect parameter and applies it to x-axis focal scaling.
- dd.js now builds projectionMatrix with aspect at startup.
- dd.js now recomputes docWidth, docHeight, and projectionMatrix inside resizeCanvas().
